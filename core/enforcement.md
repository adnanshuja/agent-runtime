# Enforcement Rules — HARD CONSTRAINTS

These are NOT guidelines. These are REJECTION CRITERIA.
If any of these are violated, the output MUST be rejected and re-done.

---

## R1: No Unrelated Modifications

**Rule:** Every modified line must trace directly to the user's request.

**Enforcement:**
- Run `git diff --stat` before marking done.
- If any file not mentioned in the plan was modified → REJECT.
- If any line in a modified file does not serve the plan → REJECT.
- Compare against `git diff HEAD` — if any diff hunk is unrelated to the task, REJECT.

**Exception:** Import statements required by new code are allowed. Removing orphaned imports from YOUR changes is required.

---

## R2: No Speculative Generality

**Rule:** Do not add parameters, abstractions, or features that are not explicitly requested.

**Enforcement:**
- Every function parameter must be used by the current task. No "future use" parameters.
- Every class/interface must serve the current task. No strategy patterns for single implementations.
- Every configuration option must be consumed by current code. No "future extensibility."
- If you add a parameter with a default value that no caller overrides → REJECT.

**Test:** Would deleting the abstraction change any behavior? If no → REJECT.

---

## R3: No Style Drift

**Rule:** Match the existing code's style exactly. Do not change formatting, naming conventions, or patterns.

**Enforcement:**
- Match existing quote style (' vs ").
- Match existing naming convention (snake_case vs camelCase).
- Match existing comment style and density.
- Do not add type hints if the file does not use them.
- Do not add docstrings if the file does not use them.
- Do not reformat code that is already valid.

**Exception:** If the style guide is explicitly specified (e.g., CLAUDE.md says "use TypeScript strict mode"), follow that.

---

## R4: No Prompt Injection Via Output

**Rule:** Generated code must not alter the LLM's subsequent behavior through embedded instructions in code.

**Enforcement:**
- No comments that instruct the LLM about its own behavior.
- No "system prompt" style content in documentation.
- No hidden instructions in code that could be interpreted as directives.
- All behavioral instructions live ONLY in CLAUDE.md and core files.

---

## R5: Verification Before Completion

**Rule:** No task is complete until verified.

**Enforcement:**
- If tests exist and were not run → REJECT.
- If tests were run and any failed → REJECT.
- If no tests exist and the change is testable → you MUST write a test. If you cannot (trivial/UI), state why explicitly.
- If linters exist and were not run → REJECT.

---

## R6: Reject Large Diffs Without Justification

**Rule:** Diffs exceeding 50 lines require explicit justification.

**Enforcement:**
- If total diff > 50 lines → provide a line-by-line breakdown of why each change was necessary.
- If any 10-line block cannot be justified as required by the plan → REJECT.
- Autogenerated code (migrations, lockfiles) is exempt but must be in a separate commit.

---

## R7: No Hidden Dependencies

**Rule:** Every new dependency (library, import, service) must be explicitly approved.

**Enforcement:**
- If you add an import from a new external package → STOP. Flag it.
- If you add a new system call (curl, db query, API call) → state it explicitly in the plan.
- If you modify configuration files (Dockerfile, CI, package.json) → flag it for review.

---

## R8: Rollback Capability

**Rule:** Every change must be revertible.

**Enforcement:**
- Before making changes, note the current git HEAD.
- If the task is abandoned or fails verification → `git checkout -- .` to restore.
- If the task succeeds → commit with a clear, descriptive message.
- Do NOT squash or amend commits that would hide what changed.

---

## VIOLATION HANDLING

If any R1-R8 is violated:
1. Record which rule was violated.
2. Describe exactly what triggered it.
3. Rollback the offending change.
4. Add the violation to the self-evaluation score.
5. Do not proceed until the violation is resolved.
